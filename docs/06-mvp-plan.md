# 06. MVP設計・実装方針

## 1. MVPの定義

> 「部屋にアバターがいて、自分の服と気になる服を着せ替えでき、買うかどうかの判断に役立つ」が成立すること。

完全再現・AI・カスタマイズは入れない。**コアループが気持ちよく回る**ことだけに集中する。

### MVP収録範囲

✅ アバター作成（パーツ数は最小: 髪10/顔テクスチャ各5前後/体型3/肌6）
✅ マイルーム表示（固定1ルーム、クローゼット・ミラーのタップ導線）
✅ 手持ち服登録（写真 + メタ + 見え方設定）/ クローゼット一覧
✅ 着せ替え（テンプレ約30種 × 色柄）
✅ 気になる服の登録・仮試着・相性確認（ルールベース）
✅ コーデ保存（スナップショット付き）
❌ AI画像処理 / 部屋カスタム / SNS / 共有 / 通知

## 2. 最大リスクと先行検証（Phase 0 でやる）

リスク順に技術スパイクを先に潰す。**UIから書き始めないこと。**

| リスク | 検証内容 | 失敗時のフォールバック |
|---|---|---|
| R1: RN+expo-glでスキニングGLBが安定動作するか | ベースボディGLB+服1着を実機表示、骨共有、30fps確認 | three.jsをWebView運用 or 2.5D化 |
| R2: 動的テクスチャ（Skia→three）が動くか | ボーダー柄を実行時生成して服に貼る | 柄をプリセットPNG同梱に縮小 |
| R3: 3Dアセット制作の工数 | Blenderでボディ+Tシャツ+パンツの最小セット制作→書き出し→表示の往復 | テンプレ数を10種に縮小 / 外注 |
| R4: GLViewスナップショット | takeSnapshotAsyncでサムネ保存 | スナップ無しでアイテム画像並べたサムネ |

## 3. 実装フェーズ

### Phase 0: 技術スパイク（〜2週）
- Expo dev clientセットアップ（prebuild, r3f/native, expo-gl, skia導入）
- R1〜R4の検証。ここで3D方式を確定する

### Phase 1: 土台（〜2週)
- Supabaseプロジェクト・マイグレーション（docs/04のSQL）・RLS・signupトリガー
- expo-router骨組み（タブ4 + モーダル）、Auth（メール + Apple/Google）
- UIキット（ItemCard, Chips, Palette, Button, BottomSheet）と taxonomy 定数

### Phase 2: クローゼット縦切り（〜2週）
- 手持ち服登録フロー（撮影→リサイズ→Storage→メタ入力→見え方設定）
- クローゼット一覧（フィルタ・検索・お気に入り）・詳細
- ※この時点ではまだ3D無しでもリリース可能な品質にする（データ層を固める）

### Phase 3: アバターと着せ替え（〜4週・最重量）
- アバター作成画面（パーツ切替・morph・顔テクスチャ合成）
- 服テンプレアセット制作（カテゴリごとに2〜5種、計約30種）← 並行で進める
- 着せ替え画面（slots store, GarmentMesh, ItemLook適用）
- コーデ保存（スナップショット→Storage→outfits）

### Phase 4: 部屋（〜2週）
- 部屋GLB制作・RoomScene・タップ導線（クローゼット/ミラー/アバター）
- ホームをルームビューに差し替え、登録完了アニメ等の演出

### Phase 5: ウィッシュリストと相性（〜2週）
- 気になる服登録・一覧・ステータス管理・「買った→クローゼット移動」
- 仮試着（着せ替え画面への検討中アイテム供給 + try_on_history）
- ルールベース相性スコア + 結果画面

### Phase 6: 磨き（〜2週）
- 空状態・エラー・オフラインキャッシュ・ローディング
- パフォーマンスチューニング（frameloop demand、アセットプリロード）
- TestFlight / 内部テスト → ストア申請

合計目安: **約3.5〜4ヶ月**（1人開発、3Dアセット制作込み。アセット外注なら短縮可）

## 4. 最初に書くべきコード（着手順）

1. `npx create-expo-app fit-collection -t tabs(typescript)` → prebuild → r3f動作確認（Phase 0）
2. `supabase/migrations/0001_init.sql`（docs/04全文）
3. `src/constants/taxonomy.ts`（全画面が依存する語彙）
4. `src/types/index.ts`（ItemLook ほか型定義 — 着せ替えエンジンとDBの契約）
5. `src/lib/supabase.ts` + 認証フロー
6. `src/components/three/AvatarModel.tsx`（ベースボディ表示 → パーツ切替 → 服bind の順に育てる）

## 5. 実装方針（守ること）

- **ItemLookがすべての中心**: 描画は ItemLook のみを見る。closet/wishlist/AIの区別を3D層に持ち込まない
- **3D層とデータ層を疎結合に**: features/* はthreeをimportしない。components/three/* はsupabaseをimportしない
- **アセットはカタログ駆動**: テンプレ追加 = GLB追加 + templates.ts 1行。コード変更不要に保つ
- **計算はインターフェース固定で差し替え可能に**: getCompatibility() / processItemImage()（将来AI化する2点）
- **デモデータを最初から用意**: シード服20着+アバター1体のfixtureを作り、3D開発をデータ登録なしで回せるようにする

## 6. 成功指標（MVPリリース後に見る）

- オンボーディング完了率（アバター作成まで）
- 1ユーザーあたり登録服数（10着超えると定着しやすい仮説）
- 仮試着実行率（ウィッシュリスト登録に対する）
- 「買った/いらない」への遷移率 = 購入判断に役立った証拠
- コーデ保存数 / 週次復帰率

## 7. 将来拡張ロードマップ

1. **AI登録補助**: 背景透過→色/カテゴリ自動判定→柄生成→シルエット推定（Edge Functions + 画像AI）
2. **AIコーデ提案・相性の高度化**: getCompatibility をAI実装に差し替え
3. **部屋カスタマイズ**: rooms.furniture jsonbを使用。家具=ガチャ/報酬で継続動機に
4. **共有**: コーデ画像のSNSシェア → コーデ投稿フィード
5. **カレンダー連携**: 今日のコーデ記録・おすすめ
6. **写真からアバター生成**: 顔写真→パーツ自動選択
